1. Executive Summary & Project Scope
The DAKKHO Student App is Bangladesh's premier polytechnic student streaming platform,
currently deployed as a web application at dakkho-student.pages.dev built with Next.js and
hosted on Cloudflare Pages. The platform serves students across 20 polytechnic technologies,
offering video courses, live classes, quizzes, certificates, and social features. The backend is
powered entirely by Cloudflare infrastructure: D1 (SQL database), KV (key-value cache),
Workers (serverless API), and R2 (object storage for videos and resources). Administrative
functions

are

managed

through

separate

panels

at

dakkho-admin.pages.dev

and

dakkho-instructor.pages.dev.
This plan details a complete native Android rewrite using Kotlin and Jetpack Compose,
matching the website's UI/UX exactly while adding significant security enhancements, offline
capabilities, and native device integrations. The application will consist of 111 unique screens
organized across 29 implementation phases, with each phase containing 10 or more granular
sub-phases. The video player (Phase 14) is specifically designed to match the reference
player.html UI with support for regular video formats (MP4, MKV, AVI, WEBM), HLS/m3u8
streaming, DASH, and YouTube embedded content via iframe/WebView. The green accent color
(#22C55E) and dark overlay design from the reference player will be faithfully reproduced in the
native ExoPlayer implementation.
The rewrite prioritizes: (1) pixel-perfect UI matching the website including glassmorphism
cards, gradient buttons, and the exact color system; (2) multi-format video playback with DRM
protection; (3) comprehensive security including root detection, certificate pinning, screenshot
blocking, and encrypted storage; (4) offline-first architecture with Room database caching; (5)
seamless backend synchronization with Cloudflare D1, KV, Workers, and R2; (6) dual push
notification channels (FCM + OneSignal); and (7) PipraPay payment integration via Chrome
Custom Tabs.

2. Existing Architecture Analysis (Website +
Backend)
2.1 Web Application
The student-facing web application at dakkho-student.pages.dev is built with Next.js 15 and
deployed on Cloudflare Pages. It uses a client-side SPA architecture where all pages are
rendered within a single DakkhoApp component that manages navigation state through a
Zustand store. The UI employs a consistent design system with sky-blue (#0EA5E9) as the
primary color, glassmorphism card effects (semi-transparent backgrounds with backdrop blur),
and gradient buttons (135deg sky-blue to deep-blue). The application features a TopBar with
logo/search/notification/avatar, a 4-tab BottomNav (Home, Explore, My Courses, Profile), and a
Sidebar drawer for additional navigation. All API calls go through a centralized fetch wrapper
that attaches the JWT Bearer token from localStorage.

2.2 Cloudflare Worker API
The backend API is served by a Cloudflare Worker using the Hono router framework, deployed
at the dakkho-api.pages.dev domain. The worker handles all CRUD operations, authentication,
file uploads (to R2), and serves as the intermediary between the client and D1/KV/R2. Key
middleware includes JWT authentication (Bearer token verification), CORS headers, rate
limiting (per-IP and per-user), and request logging. The worker also includes a scheduled
handler (cron) for cleaning up R2 attachments from resolved support tickets older than 30 days.

3.1 D1 Database (Relational Data)
Cloudflare D1 serves as the primary relational database, storing all structured application data.
Key tables include: users (id, email, full_name, institute_id, technology, avatar_url, role,
created_at), courses (id, title, description, instructor_id, technology, price, thumbnail_url,
is_published), enrollments (id, user_id, course_id, package_id, enrolled_at, progress),
watch_history

(id,

user_id,

video_id,

course_id,

position,

duration,

last_watched_at),

notifications (id, user_id, type, title, body, action_url, is_read), support_tickets (id, user_id,
category, status, subject, created_at), payments (id, user_id, course_id, order_id, amount,
status, pp_id, created_at), and achievements (id, user_id, badge_type, earned_at). The D1
database is accessed exclusively through the Cloudflare Worker API, never directly from the
Android app.

3.2 KV Namespace (Key-Value Cache)
Cloudflare KV is used for frequently accessed configuration data that does not require relational
queries. The KV namespace stores: site configuration (maintenance mode flag, featured course
IDs, announcement banners), payment configuration (PipraPay API keys, supported methods),
feature flags (enable_live_classes, enable_discussions, enable_social_features), and cached API
responses for popular endpoints (instructor lists, technology lists). The Android app accesses
KV data through the Worker API at /api/config, which reads from KV and returns JSON. Local
caching in Room ensures this data is available offline.

3.3 Cloudflare Workers (Serverless API)
The Worker serves as the entire backend API layer, handling authentication, CRUD operations,
file management, and business logic. Built with the Hono framework, it supports middleware
for JWT verification, CORS, rate limiting, and request validation. The Worker connects to D1 for
structured queries, KV for configuration reads, and R2 for file operations. The scheduled
handler (cron trigger) runs daily to clean up expired R2 attachments from resolved support
tickets older than 30 days. The Android app communicates exclusively with the Worker API via
Retrofit, never directly accessing D1, KV, or R2.

3.4 R2 Object Storage
Cloudflare R2 stores all binary objects: video files (course lectures in MP4/HLS segments),
course thumbnails and instructor avatars, PDF resources and slides, and support ticket
attachments. Video streaming uses R2 presigned URLs with time-limited access tokens
generated by the Worker at /api/video/stream/session/:id. The Android app never accesses R2
directly; all file URLs are obtained through the Worker API. Downloaded videos are encrypted
locally using AES-256 before storage on the device.

4. Admin Panel & Instructor System
4.1 Admin Panel (dakkho-admin.pages.dev)
The admin panel provides full platform management capabilities. Key functions include: course
management (CRUD operations, publish/unpublish, pricing), user management (view students,
block/unblock, view activity), financial management (payment history, refund processing,
revenue dashboard), content moderation (review Q&A;, discussion posts, support tickets),
system configuration (feature flags via KV, maintenance mode, announcement banners), and
analytics (enrollment trends, popular courses, revenue charts). The admin panel shares the
same Worker API with elevated permissions based on the user's role field in D1.

4.2 Instructor Panel (dakkho-instructor.pages.dev)
The instructor panel allows course creators to manage their content. Key functions include:
course creation and editing (upload videos, set curriculum, add resources), student interaction
(answer Q&A;, post announcements, grade assignments), live class scheduling (create and
manage live sessions), earnings dashboard (view course sales, withdrawal history), and profile
management (update bio, avatar, social links). Instructors authenticate through the same
/api/auth/login endpoint with role-based access control. The Android app syncs instructor data
through /api/instructors endpoints.

5. Native Android Tech Stack & Architecture
The application follows Clean Architecture with three distinct layers: Presentation (Compose
screens, ViewModels, UI state classes), Domain (models, use cases, repository interfaces), and
Data (Retrofit API services, Room DAOs, repository implementations). Dependency injection
uses Hilt with @HiltViewModel for ViewModels and @Inject for use cases and repositories. All
network calls go through Retrofit with OkHttp interceptors for authentication, logging, and
error handling. Room serves as the offline cache with Flow for reactive data streams. The
navigation system uses Jetpack Navigation Compose with type-safe @Serializable route
definitions.

6. Theme System & UI Design
The theme system faithfully reproduces the website's design language in Jetpack Compose. The
Material3 ColorScheme maps the website's CSS custom properties: primary (#0EA5E9 sky-blue),
secondary (#1E3A8A deep-blue), surface (#F8FAFC light / #0F172A dark). Glassmorphism cards
use Modifier.graphicsLayer with RenderEffect.createBlurEffect for the backdrop blur,
combined with semi-transparent backgrounds. The gradient system defines linear gradients at
135 degrees from sky-blue to deep-blue for buttons and hero sections. Typography uses the Noto
Sans SC font family with a scale matching the website's Tailwind classes. Dark mode is fully

supported with dynamic color (Material You) on Android 12+ devices.

8. Authentication & Security Framework
The authentication system follows the same JWT-based flow as the web application with
significant native security enhancements. Login uses /api/auth/login with email/password,
returning a JWT token stored in EncryptedSharedPreferences. Signup is a 4-step wizard:
credentials, institute selection, technology selection, and OTP verification. BiometricPrompt
enables quick login for returning users. The security framework includes: FLAG_SECURE on
video and payment screens to prevent screenshots, certificate pinning for Cloudflare Worker
hostname, root detection via Play Integrity API, R8 obfuscation with string encryption,
anti-tampering via APK signature verification, EncryptedSharedPreferences for all sensitive
data, overlay detection to prevent clickjacking, and network security config to prevent cleartext
traffic.

14. Video Player — Detailed UI Specification
(Reference: player.html)
This section provides the complete UI specification for the video player, based on the reference
player.html file. The player must support regular video formats (MP4, MKV, AVI, WEBM, etc.),
HLS/m3u8 streaming, DASH, and YouTube embedded content via iframe/WebView. All
hardcoded mock data from the reference is ignored; only the UI layout, icon positions, and
visual effects are replicated.

14.1 Multi-Format Playback Support
• Regular Video (MP4, MKV, AVI, WEBM, etc.): ExoPlayer DefaultMediaSource with progressive
download. Supported codecs depend on device hardware decoder. MKV support via ExoPlayer's
Matroska extractor.

• HLS (m3u8): HlsMediaSource with adaptive track selection. Quality auto-switching based on
bandwidth. Manual quality override via settings dropdown.

• DASH (mpd): DashMediaSource for DASH-compliant streams. Adaptive bitrate and quality
selection.

• YouTube Embedded Video: WebView container with YouTube IFrame Player API. Custom
controls overlay matching player.html style. Prevent YouTube app takeover (handle Intent
resolution). Fullscreen toggle via WebView reload with fullscreen parameter.

• Auto-Detection: Source type determined from URL extension (.mp4, .m3u8, .mpd) or API
response mime type. YouTube detected from youtube.com/youtu.be domain patterns.

14.2 Player UI Layout — Top Bar
The top bar uses a dark gradient overlay (rgba(0,0,0,0.42) at top fading to transparent). Left side
contains the back button (Lucide chevron-left icon, 24px, 8dp padding) and the video title (14sp
semi-bold white, max-width 320dp, ellipsis truncation). Right side contains 5 icon buttons in a
row, each 24px Lucide icon with 8dp padding and tooltip: (1) Audio/DUB button
(book-headphones icon), (2) CC/Subtitles button (closed-caption icon), (3) Bookmark button
(bookmark icon, fills white when active), (4) Episodes/Chapter list button (list-video icon,
highlighted green when panel is open), (5) Settings button (settings-2 icon). All buttons have

hover scale (1.1x) and active scale (0.88x) animations.

14.3 Player UI Layout — Center Controls
Center controls are horizontally centered with 48dp gap between each. Three buttons: (1)
Rewind 10s — circular button with FontAwesome arrow-rotate-left icon (38px) and "10" label
overlaid (12sp DM Mono bold), (2) Play/Pause — large SVG icon (68x68dp) with play triangle and
pause bars, ripple effect animation on click (green #22C55E expanding circle), (3) Forward 10s —
circular button with FontAwesome arrow-rotate-right icon (38px) and "10" label overlaid. Skip
actions trigger a "seek flash" animation showing the skip direction icon and label (+10s or -10s)
that fades out over 650ms.

14.4 Player UI Layout — Bottom Bar
The bottom bar has two sections. Upper section (right-aligned): PiP button (Lucide
picture-in-picture, 22px) and Fullscreen button (Lucide maximize when not fullscreen,
minimize when fullscreen, 22px). Lower section: Seek bar with green (#22C55E) progress fill on
a semi-transparent white track (4dp height, expands to 6dp on hover/touch). Thumb is a white
circle (16dp) with green border (2.5dp) and glow shadow (rgba(34,197,94,0.45)), visible only on
hover/touch at 1.3x scale. Time tooltip appears above seek bar on hover showing formatted time
(DM Mono, 11sp). Below seek bar: time display with current time (left) and total duration (right)
in DM Mono font (12sp, white 78% opacity).

14.5 Player UI Layout — Dropdown Panels
Three dropdown panels appear below their respective top-bar buttons, positioned at
bottom-right of the triggering button. Each dropdown has a dark background (#1C1C1E), 14dp
rounded corners, and scale+fade entrance animation (0.94x -> 1.0x, opacity 0 -> 1, 200ms). Audio
Track dropdown (DUB): section label "Audio Track" (10sp bold, #777), selectable items with
green checkmark for selected (Japanese Original, English Dub, etc.). Subtitles dropdown (CC):
section label "Subtitles" with Off/English/Bengali/Japanese options. Settings dropdown
(scrollable, max-height dynamically set): Playback Speed section (0.25x through 2x, "1x Normal"
default selected), Quality section (Auto/1080p HD/720p/480p/360p), Loop toggle ("Loop Episode"
with toggle checkmark). All selections show toast notification confirming the change.

14.6 Player UI Layout — Episode/Chapter Panel
The episode panel slides in from the right side (380dp width on tablet, full screen on phone
under 600dp width). When open, the video area shrinks to accommodate the panel. Panel
header: "Episodes" title (19sp bold) and close button (Lucide x icon, 18px). Episode list:
scrollable list with episode items. Each item has a thumbnail (116x68dp, 8dp rounded corners,
with play overlay on hover), EP number label (11sp semi-bold, green uppercase), episode title
(13sp bold, single line ellipsis), and description (11.5sp, 2-line clamp). Active episode has green

left border (3dp) and light green background (#F0FDF4). Alternate rows have slightly different

14.7 Player UI Layout — Visual Effects
• Dark Overlay: Linear gradient overlay (top 42% black -> middle 5% -> bottom 48% black) that
fades in (300ms) when controls are visible, fades out when hidden.

starts.

• Buffering Spinner: Green (#22C55E) ring spinner (46dp, 3dp border with green top), centered,

from left/right edge, scales 1.1x -> 0.9x while fading out over 650ms.

• Controls Auto-Hide: Controls overlay hides after 3 seconds of no interaction during playback.
Any touch/mouse movement shows controls immediately. Hidden state: opacity 0 with
pointer-events none.

• Toast Notifications: Rounded pill (28dp radius, rgba(10,10,10,0.9) with blur backdrop), centered
horizontally, positioned 90dp from bottom, shows for 2200ms with slide-up entrance.

18. Testing Strategy
The testing strategy follows the testing pyramid: 70% unit tests, 20% integration tests, 10% UI
tests. Unit tests cover ViewModels, Use Cases, Repositories, and Mappers using JUnit 5 and
MockK for mocking. Integration tests verify Room database operations, Retrofit API calls (using
MockWebServer), and WorkManager tasks. UI tests use Compose Testing framework to verify
screen rendering, user interactions, and navigation flows. End-to-end tests cover critical paths:
signup, login, course enrollment, video playback, payment flow, and certificate generation.
Performance profiling uses Android Profiler for memory leaks, baseline profiles for startup
time, and benchmark library for critical path measurement.

19. Parallel Execution Strategy
The 29 phases can be parallelized using shared brain (subagent) delegation. The dependency
graph allows the following parallel groups: Phase 1-3 must be sequential (foundation). After
Phase 3, Phases 4-6 (Auth) can run in parallel. Phases 7-9 (Main A) run in parallel, then 10-12
(Main B) in parallel. Phase 13 (Course Detail) must complete before Phase 14 (Video Player).
After Phase 13, Phases 15-17 (Course sub-pages), 18-19 (Instructor), 20-22 (User Activity), 23-24
(Department/Semester), and 25-27 (Profile/Settings) can all run in parallel. Phase 28
(Help/Exam) and Phase 29 (Social/Payment/Security) are the final sequential phases. With 2-3
developers or subagents working in parallel, the total timeline reduces from 15 weeks to
approximately 8-10 weeks.
